Working with AI: From Tasks to Systems

Level 4 · Architect — AI 운영체제 설계하기

이 책의 목표: 개별 워크플로우를 넘어, 워크플로우들이 서로 연결되고, 지식을 공유하며, 함께 성장하는 하나의 운영체제(AI OS)를 설계하는 것입니다.

📋 Learning Outcomes

이 책을 끝마치면 당신은 이것을 할 수 있습니다.

번호학습 목표
1여러 워크플로우를 하나의 시스템(OS)으로 통합할 수 있다.
2워크플로우 간 자동 핸드오프(Handoff)를 설계할 수 있다.
3운영 데이터가 지식으로 누적되는 체계를 구축할 수 있다.
4AI OS 전체에 적용되는 단일 거버넌스(Governance)를 설계할 수 있다.
5시스템의 전체 아키텍처를 시각화(Architecture Diagram)할 수 있다.
6오류가 발생해도 스스로 복구하는 자가 치유(Self-healing) 체계를 설계할 수 있다.

📌 Module 1: 워크플로우는 고립되어 있지 않다

① Experiment — 고립된 섬

당신이 해야 할 일: 아래 두 회사를 비교해보세요.

회사 A: 각 부서가 독립적으로 일합니다. 영업팀의 데이터는 마케팅팀에 전달되지 않고, 고객 피드백은 제품팀에 도달하지 않습니다.
회사 B: 모든 부서가 데이터를 공유합니다. 영업팀의 고객 정보는 마케팅팀으로, 고객 피드백은 제품팀으로 자동 전달됩니다.

질문: 1년 후, 더 나은 결정을 내리는 회사는 어느 쪽일까요?

② Reflection — 생각해보기

  • 당신이 만든 워크플로우들은 서로 데이터를 공유하고 있나요?
  • 한 워크플로우의 결과물이 다른 워크플로우의 재료가 되고 있나요?

③ Discovery — 방금 깨달은 것

"아, 워크플로우가 서로 연결되어야 시스템이 되는구나."

④ Rule — 법칙

법칙내용
Rule 55워크플로우는 함께 작동한다. 고립된 워크플로우는 단순한 자동화에 불과합니다.
Rule 56시스템의 가치는 연결에 있다. 워크플로우 간 핸드오프(Handoff)가 시스템의 진정한 힘입니다.
Rule 57통합은 데이터 흐름부터 시작하라. 파일, 메시지, 상태가 시스템 내에서 어떻게 이동하는지 설계하세요.

⑤ Practice — 워크플로우 간 연결점 찾기

Step 1. Level 2에서 만든 워크플로우들을 모두 나열하세요. (예: 고객 응대 워크플로우, 보고서 생성 워크플로우, 회의 준비 워크플로우)

Step 2. 각 워크플로우의 결과물(Output)이 다른 워크플로우의 재료(Input)가 될 수 있는지 확인하세요.

워크플로우 A (결과물)→워크플로우 B (재료)
고객 응대 기록→주간 보고서 생성
회의록→실행 계획 수립
데이터 분석 결과→마케팅 자료 생성

Step 3. 연결되지 않은 워크플로우가 있다면, 연결할 방법을 생각해보세요.

⑥ Common Mistakes — 하지 마세요

❌ 하지 마세요✅ 이렇게 하세요
"각 워크플로우는 독립적으로 완벽해야 해."개별 완벽함보다 시스템의 연결성이 더 중요하다.
연결을 수동으로 하려 한다.자동 핸드오프를 설계하라. (Module 2에서 배움)

  2. 각 워크플로우를 원(Circle)으로 그리고, 화살표로 연결하세요.
  3. 연결되지 않은 워크플로우가 있다면, 어떤 데이터가 흘러야 하는지 적어보세요.
  4. 이 지도를 AI_Work/architecture/workflow_map.md로 저장하세요.

📌 Module 2: 워크플로우 간 자동 핸드오프

① Experiment — 수동 전달의 한계

당신이 해야 할 일: 아래 두 가지를 비교해보세요.

상황 A: 한 워크플로우가 끝나면, 사람이 결과물을 복사해서 다른 워크플로우에 붙여넣습니다.
상황 B: 한 워크플로우가 끝나면, 결과물이 자동으로 다른 워크플로우의 입력 폴더로 이동합니다.

질문: 상황 A에서 사람이 실수로 파일을 잘못된 폴더에 넣으면, 어떤 일이 발생하나요?

② Reflection — 생각해보기

  • 당신은 워크플로우 간 데이터를 수동으로 옮기고 있나요?
  • 그 과정에서 실수가 발생한 적이 있나요?

③ Discovery — 방금 깨달은 것

"아, 수동 핸드오프는 시스템의 가장 약한 고리구나."

④ Rule — 법칙

법칙내용
Rule 58핸드오프는 자동화하라. 사람이 파일을 옮기는 단계는 오류의 온상입니다.
Rule 59폴더 구조가 핸드오프의 파이프라인이다. inbox/, processing/, done/, outbox/ 같은 폴더를 활용하세요.
Rule 60핸드오프는 상태를 전달한다. 파일만 옮기는 것이 아니라, 처리 상태(진행 중, 완료, 오류)도 함께 전달되어야 합니다.

⑤ Practice — 핸드오프 파이프라인 설계하기

Step 1. AI_Work/architecture/handoff_design.md 파일을 만드세요.

Step 2. 아래 템플릿을 채우세요.

```markdown
# 워크플로우 간 핸드오프 설계

## 폴더 구조
```

AI_Work/
├── inbox/              # 새 작업이 들어오는 곳
├── processing/         # 현재 처리 중인 작업
├── done/               # 완료된 작업 (최종 결과물)
├── outbox/             # 다른 워크플로우로 전달할 작업
└── archive/            # 오래된 완료 작업 보관

```

## 핸드오프 규칙
- `inbox/`에 파일이 추가되면 → 워크플로우 A가 자동 실행
- 워크플로우 A 완료 → 결과물을 `outbox/`로 이동
- `outbox/`에 파일이 추가되면 → 워크플로우 B가 자동 실행
- 워크플로우 B 완료 → 결과물을 `done/`로 이동
```

Step 3. 이 규칙을 AI에게 알리고, 자동화하세요. (Level 1의 스킬과 Level 2의 파이프라인을 결합)

⑥ Common Mistakes — 하지 마세요

❌ 하지 마세요✅ 이렇게 하세요
핸드오프를 수동으로 유지한다.자동화할 수 있다면 반드시 자동화하라.
상태(Status)를 전달하지 않는다.파일명에 [진행중], [완료], [오류]를 포함하라.

⑦ Mini Project (20분) — 핸드오프 자동화 구현

목표: Module 1에서 그린 워크플로우 지도에서 1개의 핸드오프를 실제로 자동화합니다.

  1. Module 1의 워크플로우 지도에서 가장 간단한 연결 1개를 선택하세요.
  2. outbox/ 폴더를 만들고, 워크플로우 A가 완료되면 결과물을 이 폴더에 저장하도록 수정하세요.
  3. 워크플로우 B가 outbox/를 감시하도록 설정하세요. (또는 AI에게 "이 폴더에 새 파일이 생기면 자동으로 처리해줘"라고 명령하세요)
  4. 실제로 실행해보고, 정상 작동하는지 확인하세요.

🎯 성공 기준: 파일 하나를 넣었을 때, 두 개의 워크플로우가 연속으로 실행되어 최종 결과물이 나오면 성공입니다!

⑧ Summary — 오늘의 정리

법칙핵심 문장
Rule 58핸드오프는 자동화하라
Rule 59폴더 구조가 파이프라인이다
Rule 60상태도 함께 전달하라
오늘의 실천1개의 핸드오프 자동화

📌 Module 3: 지식이 누적되는 시스템

① Experiment — 기억하지 않는 조직

당신이 해야 할 일: 아래 두 조직을 비교해보세요.

조직 A: 매번 같은 실수를 반복합니다. "작년에 우리가 이 문제를 어떻게 해결했지?"라고 물어도 아무도 기억하지 못합니다.
조직 B: 모든 결정과 그 이유를 기록합니다. 새로운 문제가 발생하면, 과거의 유사한 사례를 참고합니다.

질문: 1년 후, 더 현명해진 조직은 어느 쪽일까요?

② Reflection — 생각해보기

  • 당신의 워크플로우는 과거의 경험을 기억하고 있나요?
  • 같은 문제가 반복될 때, 이전 해결책을 참고할 수 있나요?

③ Discovery — 방금 깨달은 것

"아, 시스템도 기억해야 하는구나. 그래야 점점 똑똑해지지."

④ Rule — 법칙

법칙내용
Rule 61시스템은 기억해야 한다. 매번 같은 문제를 새로 해결하지 않도록, 과거의 해결책을 저장하세요.
Rule 62지식은 구조화하여 저장하라. knowledge/ 폴더에 문제-해결책 쌍(Problem-Solution Pair)을 누적하세요.
Rule 63AI가 지식을 활용하게 하라. 새로운 문제가 발생하면, AI에게 먼저 과거 지식을 검색하게 하세요.

⑤ Practice — 지식 베이스 구축하기

Step 1. AI_Work/knowledge/ 폴더를 만드세요.

Step 2. 아래 템플릿으로 problem_solution_template.md 파일을 만드세요.

```markdown
# 문제-해결책 기록

## 문제 (Problem)
[발생한 문제를 설명하세요]

## 원인 (Root Cause)
[문제의 근본 원인]

## 해결책 (Solution)
[적용한 해결 방법]

## 적용 결과 (Result)
[해결 후 어떤 변화가 있었나요?]

## 관련 워크플로우
[이 문제가 발생한 워크플로우]
```

Step 3. 앞으로 문제가 발생할 때마다 이 템플릿을 채우는 습관을 들이세요.

Step 4. AI에게 말하세요: "knowledge/ 폴더를 읽고, 비슷한 문제가 있었는지 먼저 검색해줘."

⑥ Common Mistakes — 하지 마세요

❌ 하지 마세요✅ 이렇게 하세요
"기록하는 게 귀찮아."오늘 5분 기록이 내일 1시간을 절약한다.
지식을 비정형으로 저장한다.구조화된 템플릿을 사용해야 AI도 활용할 수 있다.

📌 Module 4: 단일 거버넌스 (Global Governance)

① Experiment — 규칙이 다른 부서

당신이 해야 할 일: 아래 두 회사를 비교해보세요.

회사 A: 각 부서마다 다른 규칙이 있습니다. 영업팀은 A 규칙, 마케팅팀은 B 규칙, 제품팀은 C 규칙을 따릅니다. 혼란만 가중됩니다.
회사 B: 모든 부서가 동일한 회사 규정을 따릅니다. 새로운 직원도 쉽게 적응합니다.

질문: 어느 회사가 더 일관된 품질을 유지할까요?

② Reflection — 생각해보기

  • 당신의 각 워크플로우는 각자의 규칙을 가지고 있나요?
  • 모든 워크플로우에 공통으로 적용되는 단일 규칙(Global Rule)이 있나요?

③ Discovery — 방금 깨달은 것

"아, 시스템 전체를 관통하는 하나의 규칙이 필요하구나."

④ Rule — 법칙

법칙내용
Rule 64모든 워크플로우는 하나의 거버넌스를 따른다. 파일명 규칙, 보안 정책, QC 기준이 시스템 전체에 일관되게 적용되어야 합니다.
Rule 65거버넌스는 AI_Work/rules/global/에 저장하라. 각 워크플로우의 개별 규칙보다 상위에 있는 글로벌 규칙을 정의하세요.
Rule 66변경은 중앙에서 관리하라. 규칙을 변경할 때는 모든 워크플로우에 일관되게 반영되도록 하세요.

⑤ Practice — 글로벌 규칙 생성하기

Step 1. AI_Work/rules/global/ 폴더를 만드세요.

Step 2. GLOBAL_RULES.md 파일을 만들고, 아래 내용을 채우세요.

```markdown
# AI OS 글로벌 규칙 (v1.0)

## 1. 파일명 규칙 (모든 워크플로우)
- 형식: `YYMMDD_프로젝트-내용.md`
- 버전: `v1.0` → `v1.1` (덮어쓰기 금지)

## 2. 보안 정책 (모든 워크플로우)
- 절대: 계정/비밀번호/API 키 저장 금지
- 절대: 실제 고객 데이터 외부 전송 금지
- 필수: 모든 외부 전송은 사람 승인 필요

## 3. 품질 기준 (모든 워크플로우)

```

Step 2. 아래 내용을 채우세요.

```markdown
# 자가 치유 정책 (Self-Healing Policy)

## 1. 재시도 정책 (Retry Policy)
- 최대 재시도 횟수: 3회
- 재시도 간격: 5초 (1회), 10초 (2회), 30초 (3회)
- 재시도 조건: 네트워크 오류, 일시적 서버 오류

## 2. 우회 경로 (Fallback)
- 주 워크플로우 실패 시: 수동 처리로 전환
- 대체 AI 모델: 주 모델이 실패하면 예비 모델 사용

## 3. 사람 호출 조건 (Escalation)
- 3회 재시도 실패 시 → S2 장애 티켓 자동 생성
- 티켓 생성 시: 오류 로그, 재시도 기록, 제안된 해결책 포함

## 4. 복구 기록 (Recovery Log)
- 모든 재시도와 우회 경로 실행은 `recovery_log.md`에 기록
```

Step 3. AI에게 "이 자가 치유 정책을 모든 워크플로우에 적용해줘"라고 말하세요.
