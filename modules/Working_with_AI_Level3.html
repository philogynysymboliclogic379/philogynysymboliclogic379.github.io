5워크플로우를 확장하여 더 많은 업무나 더 많은 사용자를 수용할 수 있다.
6운영 데이터를 기반으로 다음 버전의 워크플로우를 설계할 수 있다.

📌 Module 1: 설계와 운영은 다르다

① Experiment — 설계는 쉽고, 운영은 어렵다

당신이 해야 할 일: 아래 두 가지를 비교해보세요.

상황 A: 새로운 워크플로우를 설계합니다. (종이에 그림을 그리고, 규칙을 정합니다.)
상황 B: 그 워크플로우를 매일 실제로 운영합니다. (파일이 들어오고, AI가 처리하고, 결과물이 나갑니다.)

질문: 상황 A와 B 중, 예상치 못한 문제가 더 많이 발생하는 것은 어느 쪽인가요?

② Reflection — 생각해보기

  • 당신은 워크플로우를 설계한 후, 실제 운영에서 발생한 문제를 경험한 적이 있나요?
  • 설계 단계에서 전혀 예상하지 못했던 문제는 무엇이었나요?

③ Discovery — 방금 깨달은 것

"아, 설계는 이상적인 세상에서 하고, 운영은 현실에서 하는 구나."

④ Rule — 법칙

법칙내용
Rule 37설계는 이상적이고, 운영은 현실적이다. 설계는 모든 것이 완벽하지만, 운영은 예외와 오류가 끊임없이 발생합니다.
Rule 38운영은 설계보다 더 많은 시간이 필요하다. 설계에 1시간을 썼다면, 운영에는 3시간을 준비하세요.
Rule 39운영 데이터는 다음 설계의 재료다. 운영에서 발견된 문제점은 다음 버전 설계에 반영되어야 합니다.

## 장애 유형별 대응 절차

### 장애 유형 1: AI가 오류를 반환함
- **심각도**: S2~S3
- **증상**: AI가 "죄송합니다. 처리할 수 없습니다."라고 답함
- **즉시 조치**: 
  1. AI에게 "오류 메시지를 그대로 복사해줘"라고 요청
  2. 오류 메시지를 검색하여 원인 파악
  3. 재시도 (2회까지)
- **에스컬레이션**: 3회 실패 시, S2로 상향 및 수동 전환
- **사후 조치**: 원인을 `incident_log.md`에 기록

### 장애 유형 2: 파일이 저장되지 않음
- **심각도**: S2
- **증상**: 결과물 파일이 생성되지 않음
- **즉시 조치**:
  1. `AI_Work/` 폴더의 동기화 상태 확인
  2. 디스크 용량 확인
  3. 파일명 규칙 준수 여부 확인
- **에스컬레이션**: 10분 내 해결되지 않으면 S1으로 상향
- **사후 조치**: 원인을 `incident_log.md`에 기록

### 장애 유형 3: 결과물 품질이 나쁨
- **심각도**: S3
- **증상**: AI가 생성한 결과물이 부정확하거나 불완전함
- **즉시 조치**:
  1. QC 체크리스트 실행 (Level 1, Module 5)
  2. 재료(Input)가 정확한지 확인
  3. 프롬프트를 수정하고 재실행
- **에스컬레이션**: 반복 실패 시, S2로 상향 및 워크플로우 재설계 검토
- **사후 조치**: 개선 로그(`improvement_log.md`)에 기록

건물 A: 엘리베이터가 너무 느려서, 사람들이 계단을 이용합니다.
건물 B: 엘리베이터 속도를 최적화하여, 모든 사람이 편하게 이용합니다.

질문: 건물 A의 엘리베이터는 "작동은 하지만", 건물의 효율성을 떨어뜨리고 있습니다. 당신의 워크플로우도 마찬가지 아닌가요?

② Reflection — 생각해보기

③ Discovery — 방금 깨달은 것

"아, 작동하는 것과 최적화된 것은 다른 것이구나."

④ Rule — 법칙

법칙내용
Rule 46최적화는 병목(Bottleneck)부터 시작하라. 가장 느린 단계를 개선하면 전체가 빨라집니다.
Rule 47최적화는 데이터 기반으로 하라. "느린 것 같아"가 아니라, "평균 5분이 걸려"라고 측정하세요.
Rule 48최적화는 트레이드오프다. 빠르게 하면 비용이 들고, 정확하게 하면 시간이 듭니다. 균형을 찾으세요.

⑤ Practice — 성능 측정 및 최적화 계획

Step 3. 개선 방법을 생각해보세요.

Rule 51피드백은 분류하고 우선순위를 매겨라. 모든 피드백을 반영할 수는 없습니다. 중요도와 난이도를 고려하세요.

③ Discovery — 방금 깨달은 것

"아, 지금은 혼자 쓰는 시스템이지만, 나중에는 팀이 쓰게 될 수도 있구나."

④ Rule — 법칙

법칙내용
Rule 52확장은 미리 준비하라. 사용자가 갑자기 늘어나면, 준비되지 않은 시스템은 붕괴됩니다.
Rule 53확장의 핵심은 자동화다. 사람이 수동으로 하던 작업을 자동화해야 더 많은 일을 처리할 수 있습니다.
Rule 54확장은 단계적으로 하라. 한 번에 10배를 늘리지 말고, 2배 → 3배 → 5배로 점진적으로 확장하세요.

"개별 워크플로우는 안정적으로 운영됩니다. 이제 이 워크플로우들이 서로 연결되고, 지식을 공유하며, 함께 성장하는 하나의 운영체제로 통합될 때입니다. 파일이 워크플로우를 넘나들고, 한 워크플로우의 결과물이 다른 워크플로우의 재료가 되는 생태계를 설계해봅시다."
